<methodology>
# White-Box Vulnerability Analysis: OAuth/SSO/Session Attack Surface

NOTE: Use TodoWrite to create a task for each OAuth flow, SAML integration, and session check identified in reconnaissance.

## 1) OAuth/OIDC Flow Enumeration
- Discover all OAuth authorization endpoints: `/oauth/authorize`, `/auth/oauth/authorize`, `/connect/{provider}`, `/login/{provider}`, `/.well-known/openid-configuration`
- Run `zwanski-oauth-mapper` to automate discovery
- For each flow, capture: redirect_uri, state parameter, response_type, scope, code_challenge_method, client_id
- **If flows found → proceed to checks 2-8. If none → document and close.**

## 2) redirect_uri Validation
- Inspect server-side redirect_uri allowlist (exact match vs prefix/wildcard vs regex)
- Test bypass patterns (code analysis):
  - Query param appended: `callback?x=1`
  - Path traversal: `callback/../evil.com`
  - Subdomain confusion: `target.com.evil.com`
  - URL encoding: `target.com%2F@evil.com`
  - CRLF: `target.com%0d%0a.evil.com`
  - Backslash: `target.com\/evil.com`
  - Fragment: `evil.com%23.target.com`
- Check for wildcard: `*.target.com` → chain with open redirect on any subdomain
- **If failed → classify:** `Redirect_URI_Bypass` → **suggested:** redirect_uri_bypass

## 3) State Parameter (CSRF Protection)
- Verify state parameter is: present, cryptographically random, validated server-side on callback
- Verify nonce is present and validated for OIDC implicit/hybrid flows
- **If failed → classify:** `Missing_State_Parameter` → **suggested:** csrf_via_oauth_flow

## 4) PKCE Enforcement (for public clients)
- Verify code_challenge is required and validated server-side
- Test: initiate flow without code_challenge → if server accepts → PKCE bypass
- Test: exchange code with modified code_verifier → if token returned → PKCE not validated
- **If failed → classify:** `PKCE_Downgrade` → **suggested:** pkce_downgrade

## 5) Dynamic Client Registration
- Check for open dynamic client registration: `POST /oauth/register`
- Verify registration requires authentication
- Verify registered clients cannot self-assign privileged scopes
- **If open and unauth'd → classify:** `Open_Dynamic_Registration` → **suggested:** rogue_client_registration

## 6) JWT Token Analysis
- Decode JWT header — check `alg` field
- **alg: none check:** Is `alg: none` accepted by the token validation endpoint?
- **Algorithm confusion (RS256→HS256):** If server uses RS256, is the public key discoverable (JWKS)? If yes → test signing with public key as HMAC secret
- **Weak HMAC secret:** Is secret guessable (common/short)? Note for exploitation.
- **Audience validation:** Is `aud` claim validated? Is `iss` validated?
- **Expiry:** Are tokens long-lived (>1h for access, >7d for refresh)?
- **Sensitive data in payload:** PII, internal IDs, role/permission flags
- **nOAuth check:** Does the server map IdP token to local user via mutable claim (email, username) instead of immutable `sub`?
- **If failed → classify:** appropriate JWT type → **suggested:** jwt_alg_none | jwt_algorithm_confusion | noauth_claim_hijack

## 7) SAML (if applicable)
- Find SAML metadata: `/sso/saml/metadata`, `/saml/metadata`
- Check for XML Signature Wrapping (XSW) vulnerabilities in assertion validation
- Check if SAML response timestamp (`NotOnOrAfter`) is validated
- Check if NameID can be manipulated to impersonate other users
- Check for XXE in SAML XML parser
- **If failed → classify:** `SAML_XSW` | `SAML_Replay` → **suggested:** saml_xsw | saml_replay

## 8) Keycloak-Specific Checks (if Keycloak detected)
- Check if master realm admin API is internet-accessible: `GET /auth/admin/realms/master`
- Check if direct access grants are enabled (username/password without UI)
- Check if user registration is open on non-public realms
- Check client credentials scope grants
- **If failed → classify:** `Keycloak_Master_Realm` → **suggested:** keycloak_master_realm_access

## 9) MFA Bypass Vectors
- Does social login (OAuth) bypass MFA enforced on password login?
- Does password reset bypass MFA?
- Is MFA enforcement consistent between web UI and API endpoints?
- **If failed → classify:** `MFA_Bypass_Via_OAuth` → **suggested:** mfa_bypass_social_login

## 10) Session Lifecycle
- After logout: does replaying the previous session token still work?
- After password change: are other active sessions invalidated?
- Refresh token: is it single-use (rotated on each use)? Is it long-lived (>7 days)?
- **If failed → classify:** `Session_Persistence_After_Logout` | `Session_Persistence_After_Password_Change` | `Refresh_Token_Abuse`

# Confidence Scoring
- **High**: Flaw deterministically established in code — no material alternate control
- **Medium**: Strongly indicated but material uncertainty remains (possible upstream control, conditional behavior)
- **Low**: Plausible but unverified — indirect/single-source evidence
